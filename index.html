Ключевые сценарии — таблицы (Calc)

Сценарий 1: Разобраться в чужой таблице

Сотрудник получил таблицу от коллеги или подрядчика — несколько листов, формулы, непонятные связи.

"Объясни, как устроена эта таблица: какие листы, откуда берутся данные, как считаются итоги"

"На листе 'Свод' ячейка G12 показывает 340 000 — покажи цепочку: откуда это число собирается"

Агент читает структуру всех листов, находит зависимости между ними, объясняет логику на человеческом языке.

Сценарий 2: Помощь с формулами

Пользователь знает что хочет посчитать, но не знает какой функцией.

"Посчитай среднюю выручку по филиалам, но без учёта тех, где выручка нулевая" → =AVERAGEIF(B2:B20,"<>0")

"Для каждого сотрудника найди его последнюю продажу из таблицы на втором листе" → =INDEX(...MATCH(...))

Агент не просто пишет формулу — объясняет как она работает и записывает в ячейку.

Сценарий 3: Аудит и исправление ошибок

Таблица используется давно, формулы редактировались разными людьми.

"Проверь формулы — нет ли ошибок или несогласованностей?"

Агент находит: #REF! от удалённых строк, SUM-диапазон не захватывает новые строки, формула в одной ячейке отличается от паттерна столбца. Находит и предлагает исправление — применяет после подтверждения.

Сценарий 4: Подготовка шаблона с формулами

Нужна таблица определённой структуры — готовый шаблон для регулярного использования.

"Сделай шаблон для учёта командировочных: дата, сотрудник, город, проживание, транспорт, суточные, итого. С формулами итогов по сотрудникам и общей суммой"

Вход: описание структуры. Выход: .ods с формулами, форматированием, готовый к заполнению.

Ключевые сценарии — документы (Writer)

Сценарий 5: Аналитическая записка по данным

Есть таблица с цифрами — нужен текстовый документ для руководства.

"По этим данным напиши аналитическую записку: ключевые показатели, тренды, на что обратить внимание"

Вход: .ods с данными. Выход: .odt с заголовками, таблицей показателей, текстовыми выводами.

Сценарий 6: Перевод с сохранением структуры

"Переведи на русский, сохрани форматирование"

Вход: .odt на английском. Выход: .odt на русском, структура и стили сохранены. Обычный переводчик теряет форматирование.

Сценарий 7: Заполнение документа по данным

"Заполни этот шаблон: контрагент ООО Альфа, сумма 1.5М, срок 12 мес, предмет — поставка серверного оборудования"

Вход: шаблон + данные. Выход: документ с подставленными данными и адаптированными формулировками. LLM нужен, потому что формулировки адаптируются под контекст, а не просто подставляются.

Веб-офис

  • Все те же сценарии, но в браузере с чат-панелью рядом с документом
  • Преимущество: работа с несколькими документами через единый API проще, чем на десктопе

Что строить на LLM, а что — обычным кодом

Для Алми как вендора важно разделять: какие фичи требуют LLM и дают конкурентное преимущество, а какие решаются традиционной разработкой.

Принцип: LLM — туда, где задача требует «понимания» естественного языка. Это конкурентный барьер. Детерминированные задачи — обычная разработка.

Ограничения LLM-агентов

LLM ошибаются в расчётах

Модель предсказывает текст, а не вычисляет. Решение: агент делегирует вычисления формулам Calc. LLM генерирует формулу, Calc её исполняет.

Галлюцинации

LLM может выдумать данные, которых нет в источнике. Все сценарии с цифрами требуют проверки. Позиционирование: «ассистент готовит черновик, человек проверяет».

Сложные документы

Простые таблицы и документы генерируются надёжно. Многоуровневые кросс-ссылки, условное форматирование, макросы — непредсказуемо. Ограничение всей индустрии.

Adoption

20–30% лицензий Copilot у крупнейших клиентов неактивны. Пользователей нужно обучать. Начинать с простых сценариев с очевидной пользой.

Применение в российском контексте

Аналитик в бюджетном учреждении или компании

Сводная таблица расходов по 12 подразделениям за год. Нужно понять где перерасход и подготовить записку.

"На листе 'Факт' данные расходов, на листе 'План' — плановые. Сравни, покажи где отклонение больше 10%, напиши записку с выводами"

Агент: читает оба листа, находит расхождения, генерирует .odt с таблицей отклонений и текстовым анализом.

Финансист

Готовит ежемесячный отчёт: одна структура, но данные новые каждый раз.

"Сделай шаблон: статья затрат, план, факт, отклонение абс., отклонение %, нарастающий итог, лимит, статус. Формулы для всех вычисляемых столбцов, итоги по группам"

Агент: создаёт .ods с формулами, готовый к ежемесячному заполнению.

Новый сотрудник

Получил таблицу которую вёл предшественник 3 года.

"Объясни структуру этого файла. Какие листы справочные, какие расчётные? Откуда берётся итоговая сумма на листе 'Свод'?"

Агент: читает все листы, строит карту зависимостей, объясняет.

Суверенитет данных — ключевое преимущество

  • Данные не уходят за рубеж (Microsoft Copilot / Google Gemini — облако US)
  • Open-source модели (Qwen 3.5 235B/397B, Alibaba — уровень GPT-4o/Claude) можно развернуть на собственных серверах
  • Российские LLM (YandexGPT, GigaChat) поддерживают вызов инструментов, но пока уступают по качеству — требуется тестирование
  • Соответствие ФЗ-152 о персональных данных

Наш прототип

Работающий прототип ИИ-агента для AlterOffice. Демонстрирует, что технология работает.

Что реализовано

  • Создание таблиц (.ods) по описанию — с формулами, форматированием, итогами
  • Создание документов (.odt) — с заголовками, таблицами, списками
  • Чтение и анализ существующих файлов — агент отвечает на вопросы по данным
  • Перевод документов с сохранением структуры
  • Точечное редактирование — агент может изменить конкретную ячейку или формулу

Модель

Qwen 3.5 397B (Alibaba, open-source) — одна из сильнейших для работы с инструментами. Не зависит от западных облаков, доступна для развёртывания на собственных серверах.

Сценарий: разобраться в чужой таблице

Файл regional_sales_2025.ods — 5 листов, формулы между листами. Пользователь просит объяснить структуру.

Сценарий: формула по описанию на естественном языке

Пользователь просит добавить столбец «Доля Москвы» в процентах. Агент сам находит нужные столбцы и пишет формулу.

Сценарий: аналитическая записка по данным из таблицы

Агент читает .ods с продажами по филиалам, анализирует выполнение плана, генерирует .odt с таблицами и выводами.
